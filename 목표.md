# 목표 / 계획

## 대화 분석
- 공정성(fairness)을 추가했으니, 보고서에 왜 필요했는지(무엇을 방지/개선하는지) 설명이 필요할 수 있음.
- 랜덤/간단한 RR 대비 튜닝된 정책이 항상 낫지 않다는 인식이 있어, 결과 차이는 작거나 없을 가능성.
- 현재 테스트벤치는 요청-응답 순차 구조라 공정성/성능 지표 평가에 부적합. `STEP2/SIM/TB/SAL_TB_TOP.sv`
- 공정성 스케줄러 변형에서 기본값(grant_pre/grant_ref) 누락 언급됨. `STEP2/RTL/SAL_SCHED_CAS_CNT_0120.sv`
- 성능 차이가 작으면 “시도했으나 큰 차이 없음”으로 보고 가능(근거 포함).

## 계획
- 공정성 목표와 평가 지표 정의(평균/99p 지연, 최악 지연, Jain’s fairness, throughput).
- 베이스라인 스케줄러와 공정성 버전을 확정하고, 필요 시 기본값 누락 보완 및 `script/filelist_step2.f` 반영 여부 확인.
- 여러 뱅크/ID에 동시 요청이 누적되도록 테스트벤치 확장; 지연/처리량 기록.
- 동일 workload/seed로 베이스라인 vs 공정성 비교 후 표/그래프로 정리.
- 차이가 작을 경우 원인과 함께 정리하고, 관련 레퍼런스 1~2개 인용.
